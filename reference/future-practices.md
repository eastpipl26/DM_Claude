## 1. agency-agents 서브에이전트 컬렉션 (232개 정의)

**출처**: msitarzewski/agency-agents

**무엇인가**: 다양한 역할의 subagent 정의 232개를 모아둔 오픈소스 컬렉션. 각 정의는 역할·페르소나·작업 범위를 담고 있다.

**언제 꺼내 쓰나**: 말숙이 팀(현재 19명)이 더 늘어나서 새 직원을 자주 채용하게 될 때, 재이가 새 역할을 설계하기 전에 이 컬렉션에서 비슷한 역할 패턴을 먼저 찾아보고 참고한다. 그대로 베끼지 않고, 역할 범위·책임 경계를 잡는 아이디어로만 쓴다.

**적용 시 주의**: 네이밍은 그대로 가져오지 않는다 — `@rules/naming-and-org.md`의 직원 네이밍 기준(한국 이름, 역할 비틀기)을 항상 따른다. 이 컬렉션은 "어떤 역할이 필요한지" 아이디어 소스로만 쓴다.

## 2. Research → Plan → Execute → Review → Ship 워크플로우 패턴

**출처**: 여러 Claude Code 베스트프랙티스 저장소(Superpowers, Everything Claude Code 등)에서 공통으로 발견되는 패턴

**무엇인가**: 큰 작업을 5단계로 명시적으로 나눠 진행하는 패턴.
1. **Research** — 문제·맥락 조사 (코드베이스, 기존 자료, 도메인 지식)
2. **Plan** — 구체적 실행 계획 수립, 사용자 승인
3. **Execute** — 계획대로 구현
4. **Review** — 결과 검증 (코드 리뷰, 테스트, 사실 확인 등)
5. **Ship** — 최종 반영·배포·저장

**언제 꺼내 쓰나**: AI OS 구축, 대시보드 신규 개발처럼 규모가 큰 플랜 작업을 시작할 때. 지금까지는 플랜 수립 후 바로 구현으로 넘어가는 경우가 많았는데, **Review 단계를 명시적으로 분리**하면 결과물 품질 점검이 누락되지 않는다.

**적용 예시**: 새 대시보드 만들 때 — Research(기존 dashboard 분석) → Plan(이미 있음, EnterPlanMode) → Execute(구현) → **Review(직접 띄워서 실제로 동작하는지 검증 — 지금 생략되던 단계)** → Ship(최종 커밋/배포).
