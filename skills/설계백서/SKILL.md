# 설계백서 스킬

도요님 개인 FAB 도메인 지식 축적 스킬. 현장에서 얻은 지식·이슈를 던지면, 관련 법규·기준을 검토하고
보완할 부분을 먼저 물어본 다음 md로 정리·저장한다. 업무목록과는 별개(업무 진행상황 아님, 지식 자료).

## 사용법
```
/설계백서 <내용>
```
스킬 호출 없이 "이거 설계백서에 넣어줘", "이건 업무 아니고 백서용이야" 같은 말로도 동일하게 처리한다.

## 데이터 위치
`{sk_hynix}\40_Resources\FAB기술\{주제}.md` (config.md의 `sk_hynix` 참조)

## 동작

1. **기존 파일 확인**: `40_Resources\FAB기술\` 폴더를 확인해 관련 주제 파일이 이미 있으면 해당 파일에 추가, 없으면 신규 파일 생성.
2. **검토**: 입력 내용과 관련된 법규·기준(KS, KOSHA 기술지침, 건축법 등)을 검색해 검토한다.
   - 근거를 찾으면 출처(규격번호·조항)를 명시한다.
   - 근거를 못 찾으면 "확정 근거 확인 안 됨"이라고 명시한다 — 추측을 사실처럼 쓰지 않는다.
3. **보완 질문**: 검토 결과를 바탕으로, 어떤 관점을 추가하면 정리가 더 완성도 있을지 도요님께 먼저 물어본다.
   (예: "이 내용에 A 기준·B 사례를 더 붙이면 좋을 것 같은데, 반영할까요?" — AskUserQuestion 또는 채팅으로 확인)
   사용자 확인 없이 임의로 방향을 정해 md를 완성하지 않는다.
4. **저장**: 답변 반영해 md로 요약 정리·저장. 기존 파일 스타일 따름(`CR_벽체_종류.md`, `방폭_기자재.md` 참고):
   - 제목(`# 주제명`)
   - `> FAB 설계백서용 개인 학습 정리. 업무목록과 별개, 도메인 지식 축적용.` 안내문
   - 본문(개념·기준·비교표 등)
   - 관련 업무목록 항목이 있으면 `## 관련 실무 이슈` 섹션에 링크 연결 (`../업무/items/{ID}_*.md`)
5. **보고**: 저장된 파일 절대경로 알려줌.

## 하지 말 것

- 법적 근거가 불확실한 걸 확정처럼 서술하지 않는다.
- 보완 질문 단계 없이 바로 완성된 md를 저장하지 않는다.
- MOC(전체 목차) 파일은 지금 만들지 않는다 — 주제별 파일만 축적, MOC는 나중에 일괄 생성 예정.
