# 업무-관리 스킬

SK하이닉스 본업 업무 관리. **개별 노트(items) + Obsidian Bases** 방식.

> 2026-07-20: 기존 표(md 단일 파일) + python HTML 대시보드 방식은 폐기, 관련 파일 전부 삭제됨
> (`업무목록.md`/`인허가.md`/`업무대시보드*.html`/`build_work_dashboard.py`).
> 이력은 `업무목록_구표형식_백업.md`/`인허가_구표형식_백업.md`에만 남아 있음(참고용, 갱신 안 함).
> 실제 업무는 `items/` 폴더 안에 노트 1개 = 업무 1개로 관리하고, `업무현황.base`(Obsidian Bases)로 필터링해서 본다.

## 사용법
```
/업무-관리 <자연어 지시>
```
예시:
```
/업무-관리 추적 추가: F02 CR층 도면 회신, 공사팀
/업무-관리 73번에 로그 추가: 김동진 소장 회신 요청 발송
/업무-관리 73번 완료 처리
/업무-관리 현황
```
스킬 호출 없이 "업무 하나 추가해줘", "이거 로그 남겨줘" 같은 자연어로 말해도 동일하게 처리한다.

---

## 데이터 위치 (config.md `sk_work` = `{sk_hynix}\업무`)

- `items\` — 업무 노트 폴더. 파일명: `{3자리ID}_{phase}_{키워드}.md` (예: `073_Y1P1_CR코어_EV배기_벽체오프닝_추가검토.md`)
- `인허가_items\` — 인허가 건 노트 폴더. 같은 규칙.
- `업무현황.base` — items 폴더를 상태/카테고리별로 보는 Obsidian Bases 뷰. 필터: `file.folder == "20_Projects/SK하이닉스_용인/업무/items"`
- `인허가현황.base` — 인허가_items 뷰.
- `업무가이드.md` — 액션유형/rework원인/TER·LL 판단기준과 예시. **새 item 등록·태깅 전에 이 문서 기준을 따른다.**
- `정산_Rework.base` — rework원인이 있는 건만 모아 원인별(설계오류/도면오류/시공변경요청/발주처변경요청/MEP변경요청/연쇄영향)·액션유형별로 보는 뷰. 정산 근거자료용.
- `TER_LL.base` — 카테고리에 TER 또는 LL이 태그된 건만 모아보는 뷰.
- `공사일정.md` — 표 형식 유지 중 (일정은 아직 마이그레이션 안 됨). 항목·구분(설계/공사/인허가/개인)·시작일·마감일·상태.

### item 노트 형식 (frontmatter)
```yaml
---
id: "73"
구분: "추적"          # 일일/긴급/추적/일반
제목: "Y1P1 CR코어 E/V배기 벽체오프닝 추가검토"
대응처: "정림/설비"     # 발주처/CM/공사팀/공무팀/인허가/내부/개인 등 자유기술 가능
담당자: "서동민(본인)"
마감일: "2026-07-23"   # 불명확하면 공란
완료일: ""
카테고리: ["구조", "협의", "TER"]   # 법규/간섭/설계변경/시공성/구조/비용/BIM관리/인허가/LL/TER/협의/일정 중 해당 태그. TER=Trouble Error Report(실사고·트러블), LL=Lessons Learned(교훈사례) — 업무가이드.md 3장 참고
상태: "⏳대기"          # ✅완료 / 🟡진행 / 🔴지연 / ⏸보류 / ⏳대기 / 🔵지속관리
등록일: "2026-07-15"
액션유형: ["시공확인필요", "정림회신요청"]   # 시공확인필요/공사현황확인필요/발주처보고필요/정림회신요청/CM검토조치/BIM관리/법규검토/기타 중 해당 값(복수 가능). 업무가이드.md 1장 기준
rework원인: ["시공변경요청"]   # 재작업을 유발한 경우만: 설계오류/도면오류/시공변경요청/발주처변경요청/MEP변경요청/연쇄영향/기타. 해당 없으면 [] 유지. 업무가이드.md 2장 기준
---

## 메모 로그

- [MM-DD] 내용 (조사 최대한 생략, 개조식)
```

## 동작

1. **먼저 기존 item 검색**: 같은 주제의 업무가 이미 있는지 `items/` 폴더에서 제목·키워드로 확인한다. **있으면 신규 등록하지 말고 그 노트에 메모 로그만 추가한다.**
2. **신규 등록 시에만**: `items/`에서 기존 최대 ID+1 확인 → 새 파일 생성 (frontmatter + 첫 로그 1줄). `업무가이드.md` 기준으로 `액션유형` 채우고, 재작업을 유발한 건이면 `rework원인`도 채운다(해당 없으면 `[]`). TER/LL 해당하면 `카테고리`에 추가.
3. **로그 추가**: 해당 item 파일의 "## 메모 로그" 아래에 `- [MM-DD] 내용` 한 줄 추가 (기존 로그는 지우지 않고 이어붙임). 진행 중 rework 성격이 새로 드러나면 `rework원인`도 그때 채운다.
4. **완료 처리**: 상태를 `✅완료`로 변경 + 완료일 채움 (행 삭제 아님, 파일도 삭제하지 않음).
5. **재생성할 HTML 없음** — Obsidian Bases는 vault에서 실시간으로 보이므로 별도 빌드 스크립트 실행 불필요. (구 python 스크립트 `build_work_dashboard.py`는 더 이상 쓰지 않음.)
6. **보고**: 등록/갱신한 item 파일 경로 + 반영 내용 요약.
7. **현황 요청 시**: `items/` 폴더를 훑어 긴급 → 마감임박(D-7) → 지연 순으로 채팅에 요약. 파일 수정 없음.

## 회의록·이슈 연계 (자동)

회의록(meeting-notes)이나 이슈(issue-register) 정리가 끝나면, 액션아이템 중 **도요가 직접 해야 하거나 기한이 있는 것**을 위 동작 1~3에 따라 `items/`에 반영한다(기존 건이면 로그 추가, 신규면 새 노트). 로그에 출처 회의록 파일명을 남긴다.

## 하지 말 것

- `업무목록.md`/`인허가.md`/`build_work_dashboard.py`를 되살리지 않는다 — 폐기·삭제됨, 만들어달라는 요청이 없는 한 재생성 금지.
- 같은 주제 업무를 중복 등록하지 않는다 — 항상 먼저 검색.
- 완료 처리는 파일 삭제가 아니라 상태 변경 (기록 보존).
- 확실하지 않은 마감일을 추측으로 넣지 않는다 — 기한 불명이면 마감일 공란.
